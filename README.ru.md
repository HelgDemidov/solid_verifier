# SOLID Verifier Dashboard

Версия на английском: [README.md](README.md)

`solid_dashboard` — это CLI-инструмент, управляемый конфигурационным файлом, который анализирует Python-проекты на предмет соблюдения принципов SOLID и слоистой архитектуры. Он запускает пайплайн статических анализаторов, вычисляет метрики, проверяет контракты импортов и генерирует машинно-читаемый отчет в формате JSON (с возможностью в будущем создавать поверх него визуальные HTML-отчеты).

Инструмент не привязан к конкретному проекту (project-agnostic): его можно переиспользовать в различных кодовых базах на Python, а не только в проекте Scopus Search API.

---

## Основные возможности

- **Единая точка входа CLI** для запуска всего конвейера анализа целевого проекта.
- **Управление через конфигурацию** с помощью файла `solid_config.json`:
  - Определение архитектурных слоев и их префиксов модулей.
  - Определение внешних "библиотечных слоев" (БД, веб-фреймворки и т.д.).
  - Включение/отключение и настройка отдельных анализаторов.
- **Статические метрики**:
  - **Сложность и поддерживаемость** через `radon` (Cyclomatic Complexity, Maintainability Index).
  - **Связность (Cohesion)** через собственный адаптер **LCOM4** на базе модуля `ast`.
  - **Граф вызовов и мертвый код** через `pyan3`.
- **Архитектура и зависимости**:
  - **Граф импортов** на базе `grimp` (тот же движок, что и у `import-linter`).
  - **Контракты слоистой архитектуры**, проверяемые с помощью CLI `import-linter`.
  - Метрики стабильности (по Роберту С. Мартину: `Ca`, `Ce`, `Instability`) для каждого слоя.
  - Явное моделирование **сторонних библиотек** как отдельных слоев.
- **Расширяемый пайплайн**:
  - Понятный интерфейс `IAnalyzer`, явно реализуемый всеми адаптерами.
  - Подключаемые анализаторы (в планах: `llmadapter` для AI-анализа архитектуры).
- **Машинно-читаемый отчет** в файле `solid_report.log` (формат JSON) и запланированные визуальные HTML-дашборды.

## Текущая архитектура и адаптеры

Дашборд реализован как небольшой внутренний фреймворк **адаптеров**. Каждый адаптер:

- Наследуется от общего интерфейса `IAnalyzer` (`solid_dashboard.interfaces.analyzer.IAnalyzer`).
- Имеет свойство `name`.
- Реализует метод `run(target_dir, context, config) -> Dict[str, Any]`.

### Реализованные адаптеры

- `radon_adapter.py`  
  Использует `radon` для вычисления цикломатической сложности и индекса поддерживаемости. Дополнительно интегрирует `lizard` для извлечения **только** количества параметров `parameter_count` (сигнал для проверки ISP) и максимальной глубины вложенности `max_nesting_depth`. Дублирующиеся метрики Lizard (CCN/NLOC/MI) намеренно игнорируются, чтобы `radon` оставался единственным источником истины.

- `cohesion_adapter.py`  
  Собственная реализация метрики **LCOM4** без внешних зависимостей, использующая встроенный модуль `ast`. Корректно обрабатывает современные возможности Python, такие как `async`, `@property` и поля Pydantic (`__annotations__`), явно игнорируя конструкторы `__init__` для математической точности расчетов. Полностью реализует интерфейс `IAnalyzer`.

- `import_graph_adapter.py`  
  Строит граф импортов с помощью `grimp` (движок от `import-linter`). Группирует модули по слоям на основе `solid_config.json`, идентифицирует зависимости от сторонних библиотек (DIP) и рассчитывает метрики стабильности Мартина (`Ca`, `Ce`, `Instability`) для каждого слоя. Результаты синхронизированы с архитектурой, описанной в `solid_config.json`.

- `importlinteradapter.py`  
  Интегрируется с `import-linter` через стабильный CLI (`lint-imports`), а не через нестабильное внутреннее Python API. Динамически переписывает блок `layers:` во временном конфигурационном файле на основе `solid_config.json`, затем запускает CLI как подпроцесс с изолированным `PYTHONPATH` и парсит вывод в структурированный JSON (соблюденные/нарушенные контракты, ошибки, сырой текст консоли).

- `pyan3adapter.py`  
  Использует `pyan3` для построения графа вызовов, дедупликации ребер и поиска потенциального **мертвого кода**. Специфичный для FastAPI код (например, функции роутеров) отфильтровывается во избежание ложных срабатываний. Возвращает количество узлов/ребер и их списки согласно интерфейсу `IAnalyzer`.

### Запланированные адаптеры и функции

- `llmadapter.py` (в разработке)  
  AI-анализ с использованием API OpenAI/Ollama для оценки высокоуровневых принципов SOLID (OCP, LSP, ISP, SRP, DIP) на основе контекста, собранного статическими адаптерами (граф импортов, граф вызовов, связность и т.д.). Будет возвращать структурированные выводы с оценкой уверенности и критичности.

- `differ.py` & `generator.py` (в разработке)  
  Инструменты для рендеринга JSON-отчета в визуальный HTML-дашборд (шаблоны Jinja2) и отслеживания деградации или улучшения метрик с течением времени путем сравнения текущих отчетов с сохраненным baseline-файлом.

│       │       ├── differ.py                 # (В разработке) Логика сравнения отчета с baseline
│       │       └── generator.py              # (В разработке) Логика рендеринга HTML из JSON и шаблонов

## Конфигурация (`solid_config.json`)

Файл `solid_config.json` находится в корне анализируемого проекта (`scopus_search_code/`) и описывает, как инструмент должен интерпретировать структуру проекта:

- `package_root`: корневой Python-пакет для анализа (например, `app`).
- `layers`: маппинг логических слоев на префиксы модулей (относительно `package_root`), например:

  ```json
  {
    "package_root": "app",
    "layers": {
      "routers": ["routers"],
      "services": ["services"],
      "infrastructure": ["infrastructure"],
      "models": ["models"]
    },
    "ignore_dirs": ["__pycache__", ".venv", "tests"],
    "external_layers": {
      "db_libs": ["sqlalchemy"],
      "web_libs": ["fastapi", "starlette"]
    }
  }
  ```

Эти настройки используются сразу для двух целей:

- `import_graph_adapter.py` (для визуализации и расчета метрик стабильности).
- `importlinteradapter.py` (для генерации контракта `layers:`, потребляемого `lint-imports`).

## Запуск Дашборда

Существует два основных способа запустить пайплайн.

### 1. Напрямую через внутренний CLI

Из корневой директории репозитория (`scopus_search_code/`), с активированным виртуальным окружением и установленными зависимостями:

```bash
python -m tools.solid_verifier.solid_dashboard \
    --target-dir ./app \
    --config ./solid_config.json
```

Это действие:

1. Загрузит `solid_config.json` из корня проекта.
2. Последовательно запустит все настроенные адаптеры (Radon, Cohesion, Import Graph, Import Linter, Pyan3).
3. Выведет JSON-отчет в стандартный поток вывода (stdout).
4. Запишет этот же JSON в файл `tools/solid_verifier/solid_dashboard/report/solid_report.log`.

### 2. Через удобный скрипт-обертку

Из того же корня проекта выполните:

```bash
python run_solid_dashboard.py
```

Скрипт:

1. Определит `target_dir` как `./app` и `config_path` как `./solid_config.json`.
2. Вызовет внутренний CLI:

   ```bash
   python -m tools.solid_verifier.solid_dashboard \
       --target-dir ./app \
       --config ./solid_config.json
   ```

3. Пробросит код завершения процесса пайплайна выше.

Такой подход сохраняет стабильность использования даже в том случае, если внутренний путь к пакету (`tools.solid_verifier.solid_dashboard`) в будущем изменится.

## Зависимости

Основные сторонние библиотеки, используемые дашбордом:

- [`radon`](https://pypi.org/project/radon/) — метрики сложности и поддерживаемости.
- [`lizard`](https://pypi.org/project/lizard/) — только для подсчета количества параметров функций и глубины вложенности.
- [`grimp`](https://github.com/seddonym/grimp) — построение графа импортов.
- [`import-linter`](https://pypi.org/project/import-linter/) — обеспечение соблюдения архитектурных контрактов через CLI.
- [`pyan3`](https://pypi.org/project/pyan3/) — построение графа вызовов и анализ мертвого кода.
- [`jinja2`](https://pypi.org/project/Jinja2/) — генерация HTML-отчетов (в планах).
- [`httpx`] (>=0.24.0): Синхронный HTTP-клиент с гранулярной настройкой таймаутов (connect/read/write), используемый LLM-адаптером для надежного взаимодействия с API LLM-провайдеров (OpenAI, Ollama).

Все зависимости жестко зафиксированы в `requirements.txt` на стабильных версиях, совместимых с текущей средой выполнения Python.
