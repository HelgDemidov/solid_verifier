# SOLID Verifier Dashboard

Версия на английском: [README.md](README.md)

## Вступление

Идея этого анализатора кода родилась из парадокса современной ИИ-разработки: во время создания моего первого учебного проекта (Scopus Search Code) скорость генерации кода нейросетью начала опережать мою способность осознавать усложняющийся граф зависимостей. Мой аналитический бэкграунд потребовал вернуть контроль над развивающейся системой. Я задумался об инструменте, который решал бы сразу две задачи: выступал бы строгим независимым контролером архитектурной правильности и одновременно служил наглядной картой, обнажающей скрытые связи и узлы внутри кодовой базы.

Так появилась концепция SOLID-верификатора. Для объектно-ориентированного Python именно принципы SOLID остаются наиболее зрелым и универсальным эталоном качества, позволяющим оценить здоровье архитектуры. Однако в фундаменте этого выбора заложен и глубоко личный смысл. Внутреннюю красоту, логику и философию SOLID мне открыл мой друг - талантливый инженер и разработчик с многолетним опытом. Этот проект - дань уважения его мастерству и посвящение ему.

***

`solid_dashboard` — это CLI-инструмент, управляемый конфигурационным файлом, который анализирует Python-проекты на предмет соблюдения принципов SOLID и слоистой архитектуры. Он запускает пайплайн статических анализаторов, вычисляет метрики, проверяет архитектурные контракты и — опционально — углубляет анализ нарушений OCP/LSP с помощью LLM. Результат записывается в машинно-читаемый отчет в формате JSON (`solid_report.log`).

Инструмент не привязан к конкретному проекту (project-agnostic): его можно переиспользовать в различных кодовых базах на Python.

## Основные возможности

- **Единая точка входа CLI** и полная оркестрация через `pipeline.py` для запуска всего конвейера анализа целевого проекта.
- **Строгая изоляция и управление через конфигурацию** (`solid_config.json`):
  - **Единая точка контроля директорий:** целевая директория (`package_root`) и исключения (`ignore_dirs`) задаются в одном месте и гарантированно соблюдаются всеми статическими и LLM-адаптерами. Пайплайн надежно изолирован от `.venv`, тестов и служебных скриптов.
  - Определение архитектурных слоев и их префиксов модулей.
  - Определение внешних «библиотечных слоев» (БД, веб-фреймворки и т.д.).
  - Безопасное управление секретами (LLM-ключи автоматически подхватываются из файла `.env`, оставляя JSON-конфиг чистым).
- **Статические метрики**:
  - **Сложность и поддерживаемость** через `radon`: цикломатическая сложность (CC) на уровне функций/методов с рангом A–F, и индекс поддерживаемости (MI, 0–100) на уровне файлов с рангом A–C — агрегируется в `total_files`, `mean_mi` и `low_mi_count` (файлы с рангом C).
  - **Связность (Cohesion)** через собственный адаптер **LCOM4** на базе модуля `ast` (с умной фильтрацией свойств и служебных методов для защиты от ложных срабатываний).
  - **Граф вызовов и мертвый код** через `pyan3` (полностью безопасный запуск без мутаций глобального состояния среды, с двухпроходным детектором name collision и confidence-маркировкой рёбер).
- **Архитектура и зависимости**:
  - **Граф импортов** на базе `grimp`.
  - **Контракты слоистой архитектуры**, проверяемые с помощью CLI `import-linter` (через динамическую генерацию временных контрактов на лету).
  - Метрики стабильности (по Роберту С. Мартину: `Ca`, `Ce`, `Instability`) для каждого слоя.
- **LLM-анализ OCP/LSP** (реализован, опционально включается через `solid_config.json`):
  - AST-эвристики выявляют потенциальные нарушения OCP/LSP и формируют список кандидатов.
  - `LlmSolidAdapter` отправляет каждого кандидата на верификацию в LLM (через OpenRouter).
  - Двухуровневый Anti-Corruption Layer (ACL-A + ACL-B) защищает пайплайн от некорректных ответов модели.
  - Результаты LLM органично вписываются в единый формат `Finding` рядом со статическими и эвристическими выводами.
- **Расширяемый пайплайн**: понятный интерфейс `IAnalyzer`, явно реализуемый всеми статическими адаптерами.
- **Машинно-читаемый отчет** в файле `solid_report.log` (формат JSON) и запланированные визуальные HTML-дашборды.

## Текущая архитектура и адаптеры

Архитектура верификатора осознанно построена на принципе разделения труда и минимизации недетерминированности. Мы не используем LLM там, где математика справляется лучше. 
- Три из пяти принципов SOLID (**SRP**, **ISP** и **DIP**) эффективно и со 100% надежностью контролируются детерминированным статическим анализом: расчет LCOM4 и сложности выявляет перегруженные объекты, а графы импортов жестко следят за инверсией зависимостей. Статика прозрачна, работает мгновенно и не подвержена галлюцинациям, деградации моделей или сетевым сбоям. 
- Ресурсоемкая связка «эвристики + LLM» применяется точечно — исключительно для оценки принципов **открытости/закрытости (OCP)** и **подстановки Лисков (LSP)** . Только эти два принципа требуют семантического понимания архитектурных контрактов и бизнес-логики, поэтому нейросеть подключается строго в тех узлах, на которые ей прямо укажут наши статические эвристики.

### Статические адаптеры

Дашборд на уровне статического анализа реализован как внутренний фреймворк **адаптеров**, оркестрируемый центральным модулем `pipeline.py`. Каждый статический адаптер жестко подчиняется единой точке контроля директорий (`package_root` и `ignore_dirs` из `solid_config.json`):

- Наследуется от общего интерфейса `IAnalyzer` (`solid_dashboard.interfaces.analyzer.IAnalyzer`).
- Имеет свойство `name`.
- Реализует метод `run(target_dir, context, config) -> Dict[str, Any]`.

LLM-слой реализован отдельно от `IAnalyzer` и вызывается напрямую из `pipeline.py` по результатам эвристик.

- **`radon_adapter.py`**  
  Запускает два независимых subprocess-вызова `radon` к целевой директории; оба уважают `ignore_dirs` через флаг `-i` (пустые строки и строки из пробелов фильтруются до сборки команды).

  **`radon cc --json`** — формирует нормализованный плоский список записей по каждой функции/методу: имя, тип (`function`/`method`), цикломатическая сложность, ранг (A–F), номер строки и путь к файлу. Список отсортирован по убыванию сложности.
  **`radon mi --json`** — формирует поуфайловый отчёт об индексе поддерживаемости (MI 0–100, ранг A/B/C). Агрегируется в под-объект `maintainability` с полями `total_files`, `mean_mi`, `low_mi_count` (файлы с рангом C, т.е. MI < 10) и списком `files`, отсортированным по MI ASC (худшие первыми). **Сбой MI полностью изолирован:** любая ошибка subprocess или парсинга возвращает пустой `{}` в `maintainability` и никогда не прерывает CC-результат.

  Дополнительно адаптер интегрирует `lizard` исключительно для получения `parameter_count` — количества параметров функции/метода (включая `self` для методов), недоступного в стандартном выводе radon. Метрики, вычисляемые обоими инструментами (например, CC), намеренно дедуплицируются с приоритетом radon. Нормализация `parameter_count` намеренно отложена до слоя агрегации.

- **`cohesion_adapter.py`**  
  Собственная реализация метрики **LCOM4** (Lack of Cohesion of Methods 4) без внешних зависимостей — исключительно на базе встроенного модуля `ast`. Адаптер реализован в виде двух взаимодействующих компонентов: `CohesionAdapter` (основной адаптер, реализующий `IAnalyzer`) и вспомогательного `class_classifier.py` (семантическая классификация классов). Алгоритм работает в два прохода:
  - **Pass 1:** рекурсивно обходит файлы целевой директории с соблюдением `ignore_dirs`, парсит каждый файл через `ast`, строит `ClassInfo` и `MethodInfo` для каждого класса, параллельно формируя глобальный индекс определений.
  - **Pass 2:** обогащает атрибуты каждого класса из `__init__` его предков, рекурсивно обходя MRO-цепочку по индексу; при расширении атрибутов перезапускает `_MethodUsageVisitor` для актуализации связей в графе.

  Граф связности строится на основе двух видов рёбер: общих атрибутов экземпляра (`self.field`) и вызовов методов (`self.method()`, `super().method()`). Из графа стратегически исключаются: `__init__`, методы-`@property`, пустые заглушки (`pass` / `...` / `raise NotImplementedError`). Классификатор `class_classifier.py` присваивает каждому классу семантическую роль (`concrete`, `abstract`, `interface`, `dataclass`) — агрегатные метрики (`mean_cohesion_all`, `mean_cohesion_multi_method`, `low_cohesion_count`) вычисляются **только** по `concrete`-классам, чтобы интерфейсы и dataclass-модели не искажали общую картину связности. Разрешение name collision при одинаковых именах классов в разных файлах выполняется через приоритет «того же файла», а при неустранимой неоднозначности — через graceful degradation с логированием. Адаптер полностью покрыт тестами в пакете `tests/static_adapters/test_cohesion_adapter/`.

- **`import_graph_adapter.py`**  
  Строит архитектурный граф импортов на базе `grimp`, поднимая уровень анализа с модулей до слоев, определенных в `solid_config.json` (`layers`, `utility_layers`, `external_layers`, `layer_order`).  
  Для каждого слоя вычисляет метрики стабильности Мартина (`Ca`, `Ce`, `Instability`) и формирует узлы и ребра граф а, включая «библиотечные» внешние слои. Поддерживает сквозные `utility_layers`, которые участвуют в метриках и визуализации, но намеренно исключаются из SDP/SLP-проверок для предотвращения ложных нарушений. 
  
  Реализует детектор нарушений Stable Dependencies Principle (`SDP-001`) с настраиваемым допуском `sdp_tolerance` и списком явных исключений `allowed_dependency_exceptions`. Реализует детектор Skip-Layer Principle (`SLP-001`), отслеживающий прямые зависимости через несколько тиров `layer_order`; тяжесть нарушения учитывает семантику `interface_layers`.  Все нарушения возвращаются в едином формате `violations` и снабжаются диагностическим `debug_info`, подготавливая адаптер к кросс-адаптерной корреляции в Report Aggregator.

- **`import_linter_adapter.py`**  
  Проверяет соблюдение архитектурных контрактов (layered architecture) с помощью CLI-утилиты `lint-imports`, запускаемой изолированно через `subprocess`. Перед каждым запуском адаптер **динамически генерирует временный конфигурационный файл** (`.importlinter_auto_*`): читает базовый `.importlinter` через `configparser`, записывает `root_packages` из `package_root` и транслирует порядок слоёв из `layer_order` — единственного источника истины. Временный файл гарантированно удаляется после прогона в блоке `finally`.

  Текстовый вывод `lint-imports` очищается от ANSI-кодов; статистика (`kept`/`broken`) извлекается регулярным выражением с fallback по `returncode`. Результат содержит два представления нарушений: `violations` (плоский список имён контрактов — для обратной совместимости) и `violation_details` (структурированный `List[Dict]` с полями `contract_name`, `status`, `broken_imports: [{importer, imported}]` — для будущей кросс-адаптерной агрегации). Адаптер реализует `IAnalyzer` и возвращает консистентную схему из 7 полей при любом исходе выполнения.

- **`pyan3_adapter.py`**  
  Адаптер использует `pyan3` для построения статического графа вызовов (call graph) и выявления потенциально неиспользуемого кода. Ключевое архитектурное решение: запуск выполняется через `subprocess` с параметром `cwd` — без `os.chdir`, что делает адаптер безопасным и project-agnostic. Список Python-файлов собирается вручную с соблюдением `ignore_dirs`, исключая `.venv`, тесты и служебные директории. 
  
  Адаптер реализует **двухпроходную модель разбора** текстового вывода pyan3: первый проход детектирует блоки с name collision (признак — кратные `[U]`-вхождения одного имени до де-дупликации), второй строит граф и выставляет confidence-метку `"high"` / `"low"` каждому ребру. Confidence определяется исключительно по блоку-источнику: ребро получает `"low"`, если его источник помечен как suspicious; цель ребра на confidence не влияет. Узлы итогово разделяются на три категории: `root_nodes` (нет входящих рёбер, есть исходящие — точки входа), `dead_nodes` (нет ни тех, ни других — подлинно неиспользуемый код) и обычно связанные узлы графа.

### LLM-слой: эвристики и адаптер

LLM-слой теперь опирается на более интеллектуальный статический фронт: перед запуском нейросети весь проект проходит через AST-классификацию ролей классов (`ClassRole`) и уточнённый набор SOLID-эвристик для OCP/LSP.

**Эвристический слой (AST + ClassRole)**

Модуль `heuristics.py` больше не смотрит на классы «вслепую». Для каждого `ast.ClassDef` сначала вычисляется роль класса:

- `PURE_INTERFACE` — ABC/Protocol без состояния: все non-dunder методы абстрактны.
- `INFRA_MODEL` — Pydantic/SQLAlchemy модели и аналоги (InfraScore ≥ 2 по 5 структурным признакам).
- `CONFIG` — конфигурационные классы (`BaseSettings` и аналоги).
- `DOMAIN` — все остальные доменные и сервисные классы (дефолтная роль).

OCP-эвристики исключают классы с ролью INFRA_MODEL и CONFIG; все остальные (DOMAIN, PURE_INTERFACE) проходят фильтрацию. LSP-эвристики исключают абстрактные классы (ABC, Protocol) через _is_abstract_class().

**Актуальный набор LSP/OCP-эвристик**

Обновлённый набор AST-эвристик фокусируется на доменных классах и учитывает роль родителя:

| ID          | Принцип | Что детектирует (с учётом ClassRole)                            |
|-------------|---------|-----------------------------------------------------------------|
| `OCP-H-001` | OCP     | Верхнеуровневая цепочка `if/elif` с `isinstance` (≥3 ветки)     |
| `OCP-H-002` | OCP     | `match/case` как диспетчер типов                                |
| `OCP-H-004` | OCP     | Высокая CC + `isinstance` в доменных методах                   |
| `LSP-H-001` | LSP     | `raise NotImplementedError` в переопределённом методе          |
| `LSP-H-002` | LSP     | Пустой метод-заглушка в дочернем классе                        |
| `LSP-H-003` | LSP     | `isinstance` по отношению к параметру базового типа            |
| `LSP-H-004` | LSP     | Проблемный `__init__` дочернего класса относительно родителя   |

LSP-H-004 детектирует __init__ подкласса без вызова super().__init__(). Исключения: @dataclass-классы, классы из _LSP_H004_EXCLUDED_PARENTS (object, ABC, Protocol, TypedDict, NamedTuple, BaseModel), родители с ролью PURE_INTERFACE (с ограничением: интерфейсы из project_map.interfaces без source_code не верифицируются).

Также в эвристическом слое реализован **multi-signal INFRA-фильтр**: Pydantic/SQLAlchemy/Settings-классы распознаются по набору структурных признаков (декораторы `table_name`/`__tablename__`, базовые классы `Base`/`DeclarativeBase`/ `DeclarativeBaseNoMeta`, `BaseModel`, `BaseSettings`) и получают роль `INFRA_MODEL` или `CONFIG`, после чего полностью исключаются из LSP/OCP-анализа.

### LLM-анализ OCP/LSP: архитектура слоя

LLM-анализ в SOLID Verifier устроен как отдельный слой поверх статических эвристик, а не как «черный ящик». Он работает только по кандидатам, предварительно найденным AST-эвристиками, и возвращает findings в том же доменном формате, что и остальные адаптеры (`Finding` / `FindingDetails`).

Высокоуровневая цель этого слоя — не заменить статический анализ, а уточнить и объяснить эвристические подозрения на нарушения OCP и LSP: дать человеческое объяснение, конкретную рекомендацию и оценку уверенности без риска «сломать» отчет галлюцинациями модели.

#### Поток данных: уровни анализа

Архитектура SOLID-верификатора построена на строгой двухуровневой модели. Пайплайн разделяет быстрый базовый сбор метрик и глубокий контекстный анализ нейросетью.

**Уровень 1: Базовый статический анализ (Параллельный слой)**
Все классы, реализующие интерфейс `IAnalyzer`, запускаются независимо друг от друга по данным:
- **Метрические адаптеры** (Radon, Cohesion, ImportGraph, ImportLinter, Pyan3) собирают статистику, вычисляют зацепление, цикломатическую сложность и строят графы зависимостей.
- **Эвристический адаптер** (`HeuristicsAdapter`) самостоятельно парсит исходный код проекта, строит AST-дерево (`ProjectMap`) и по набору жестких правил выявляет подозрительные участки кода — «кандидатов» на нарушение OCP и LSP (`HeuristicResult.candidates`).

**Уровень 2: LLM-надстройка (Глубокий анализ)**
Запускается строго после завершения первого уровня и только если LLM включен в конфигурации. `pipeline.py` забирает AST-карту проекта и список кандидатов от эвристик, передавая их в `LlmSolidAdapter`. Для каждого кандидата слой выполняет:
- **Context Assembler**: собирает изолированный контекст (исходный код класса, его зависимости, интерфейсы).
- **Prompt Builder**: формирует промпты (system/user) на основе `.md`-шаблонов и строгой JSON-схемы ответа.
- **LlmGateway**: отправляет запрос к провайдеру (OpenRouter/OpenAI), попутно обеспечивая кэширование, контроль бюджета токенов (Token Budget) и механизм повторных попыток (Retry).
- **Response Parser (ACL-B)**: безопасно валидирует ответ модели и преобразует его в типизированные объекты `Finding`.

В финале **Report Aggregator** объединяет плоский поток статических метрик с результатами эвристического и LLM-анализа, формируя единый сводный отчет (`solid_report.log` / HTML).

```text
 УРОВЕНЬ 1: Базовые независимые адаптеры
 ┌────────────────┐ ┌────────────────┐ ┌────────────────────────────────┐
 │ Radon, Cohesion│ │  ImportGraph   │ │ HeuristicsAdapter (AST-анализ) │

         │                  │          ┌────────────────┴───────────────┐

         │                  │          ├─ Prompt Builder                │
         │                  │          ├─ LlmGateway (Cache/Budget/ACL) │

         ▼                  ▼                           ▼

             Report Aggregator → solidreport.log / HTML
```

#### Anti-Corruption Layer: два уровня защиты

LLM-слой отделен от остального кода двумя независимыми Anti-Corruption Layer'ами.

- **ACL-A (Gateway, транспортный уровень):**  
  `OpenRouterProvider._parse_success(...)` и `LlmGateway` отвечают за безопасное чтение HTTP-ответа, обработку `finish_reason`, ошибок API, таймаутов и лимитов токенов. На этом уровне гарантируется, что на вход LLM-адаптеру попадает нормализованный `LlmResponse` с полями `content`, `tokens_used`, `model`, а не «сырая» структура провайдера.

- **ACL-B (Response Parser, семантический уровень):**  
  Функция `parse_llm_response(content: str, candidate: LlmCandidate) -> ParseResult` выполняет:
  - извлечение JSON (plain / fenced code / regex-fallback'ы);
  - структурную проверку (`findings` — список, элементы — dict);
  - поэлементную валидацию через `validate_finding(raw, candidate)`;
  - сбор `ParseResult(findings, warnings, status)` со статусом `success / partial / failure`.

Любая ошибка на этих уровнях приводит к частичному или полному отказу LLM-слоя для конкретного кандидата, но **никогда не ломает весь пайплайн**: статические и эвристические findings продолжают работать как обычно.

#### Два источника истины: эвристики и LLM

После рефакторинга эвристик архитектура «двух источников истины» стала ещё более явной.

**1. Статика и эвристики — источник координат и доверия**

Статический слой (Radon, Cohesion, ImportGraph, ImportLinter) плюс AST-эвристики и `ClassRole` определяют:

- какие файлы, классы и методы вообще стоит анализировать;
- какие из них являются доменными `BEHAVIORAL` объектами, способными нарушать OCP/LSP;
- какие именно эвристики сработали (например, комбинация `LSP-H-001` + `OCP-H-001` + `OCP-H-004` для одного и того же класса).

Для каждого кандидата формируется `LlmCandidate`, который считается **источником истины для координат**:

- `file` — путь к файлу;
- `class_name` — имя класса;
- тип принципа (`OCP`/`LSP`) — базовая рамка для будущего rule (`OCP-LLM-001` / `LSP-LLM-001`);
- список сработавших эвристик (`heuristic_reasons`).

При этом сам статический слой уже самодостаточен: на уровне последнего отчёта все цели рефакторинга эвристик выполнены, даже при нулевом вкладе LLM (LLM-кандидаты были успешно отобраны, но ответы пока не участвовали в принятии решений).

**2. LLM — источник смысла и объяснений**

LLM получает только предварительно отфильтрованных кандидатов и используется как **источник смыслового содержания**:

- формулирует человекочитаемое `message` (описание проблемы);
- уточняет принцип (`details.principle`) и даёт объяснение (`details.explanation`);
- предлагает конкретные рекомендации по рефакторингу (`details.suggestion`);
- по возможности указывает метод (`details.method_name`) и даёт best-effort информацию о том, какие эвристики были подтверждены (`details.heuristic_corroboration`). 

Все LLM-findings проходят через `Response Parser` (ACL-B), который жёстко валидирует JSON и заполняет доменную модель `Finding`. Ключевые поля (`file`, `class_name`, `source="llm"`, rule, principle) всегда строятся на основе **эвристик**, а не текста ответа модели, чтобы доверенный статический слой оставался ведущим.

Результат: статические и LLM-выводы оказываются в общем списке `Finding`, но с разными «зонами ответственности»: координаты и принадлежность к SOLID-принципу задаёт статика, объяснение и рекомендации формирует LLM.

#### Таблица полей LLM-finding'а

| Поле                              | Где хранится                          | Источник по умолчанию        | Уровень реализации                                    |
|----------------------------------|---------------------------------------|------------------------------|-------------------------------------------------------|
| `rule`                           | `Finding.rule`                        | Вычисляется (`OCP-LLM-001`)  | `validate_finding` (ACL-B)                            |
| `file`                           | `Finding.file`                        | Эвристики (`candidate`)      | `validate_finding` (из `candidate.filepath`)          |
| `class_name`                     | `Finding.class_name`                  | Эвристики (`candidate`)      | `validate_finding` (из `candidate.classname`)         |

| `severity`                       | `Finding.severity`                    | LLM → нормализация           | `validate_finding` (`error/warning/info`)             |
| `message`                        | `Finding.message`                     | LLM (обязательное поле)      | `validate_finding` (`raw["message"]`)                 |
| `source`                         | `Finding.source`                      | Всегда `"llm"`               | `validate_finding` (фиксированно)                     |
| `details.principle`              | `FindingDetails.principle`            | LLM + fallback по candidate  | `validate_finding` (JSON → `OCP/LSP` → fallback)     |

| Поле | Тип | Описание |

| `interface_layers` | `List[str]` | Слои, объявленные «интерфейсными». Влияют на тяжесть нарушений в детекторе `SLP-001`: Skip-Layer через интерфейсный слой квалифицируется строже. |

#### SDP-настройки

| Поле | Тип | Описание |
|------|-----|----------|
| `sdp_tolerance` | float | Допустимый порог нестабильности при проверке Stable Dependencies Principle (SDP-001). Если нестабильность зависимого слоя превышает нестабильность зависимости менее чем на это значение — нарушение не фиксируется. Текущее значение: `0.10` (10%). |
| `allowed_dependency_exceptions` | `List[Object]` | Явные исключения из SDP-проверок. Каждый объект: `source` (слой-отправитель), `target` (имя внешнего слоя), `reason` (обоснование). Используется для легитимных случаев, когда архитектурное нарушение осознано и задокументировано (например, ORM-модели, наследующие `SQLAlchemy Base`). |

| Поле | Тип | Описание |
|------|-----|----------|
| `ignore_dirs` | `List[str]` | Глобальный список директорий, полностью исключаемых из всех адаптеров. **Гарантирует**, что ни один адаптер не выйдет за пределы бизнес-логики: виртуальное окружение, кэши, тесты, служебные инструменты, артефакты сборки — всё исключается на уровне конфигурации, не кода. |

#### Внешние зависимости

| Поле | Тип | Описание |
|------|-----|----------|

Позволяет отслеживать зависимости на `sqlalchemy`, `fastapi` и т.д. в архитектурном контексте, а не как «чёрный ящик». |

#### LLM-настройки

| Поле | Тип | Описание |
|------|-----|----------|
| `llm.enabled` | bool | Включает или полностью отключает LLM-слой. При `false` пайплайн выполняется только на статике и эвристиках — без сетевых вызовов. |

| `llm.model` | string | Идентификатор модели в формате провайдера. Для OpenRouter — `"openai/gpt-4o-mini"` или любой другой из [openrouter.ai/models](https://openrouter.ai/models). |

Кандидаты обрабатываются в порядке убывания приоритета (количество эвристических срабатываний + наличие иерархии). |
| `llm.cache_dir` | string | Путь к директории файлового кэша LLM-ответов. Кэш-ключ — SHA-256 хэш пары (промпт + опции): идентичный запрос никогда не отправляется повторно, токены не тратятся. |
| `llm.prompts_dir` | string | Путь к директории с `.md`-шаблонами промптов и `response_schema.json`. Позволяет вносить изменения в промпты без правок Python-кода. |

## Запуск дашборда

### Предварительная настройка

Из корня репозитория (`scopus_search_code/`) с активированным виртуальным окружением:

```bash
pip install -r tools/solid_verifier/requirements.txt
```

Если используется LLM-анализ, задайте переменную окружения с API-ключом:

```bash
# Windows (PowerShell)
$env:OPENROUTER_API_KEY = "sk-or-..."

# Linux / macOS
export OPENROUTER_API_KEY="sk-or-..."
```

### 1. Напрямую через CLI

```bash
python -m tools.solid_verifier.solid_dashboard \
    --target-dir ./app \
    --config ./solid_config.json
```

Это действие:

1. Загрузит `solid_config.json` из корня проекта.
2. Последовательно запустит все статические адаптеры (Radon, Cohesion, Import Graph, Import Linter, Pyan3).
3. Если `llm.enabled: true` и найдены OCP/LSP-кандидаты — выполнит LLM-анализ через OpenRouter.
4. Выведет JSON-отчет в stdout и запишет его в `tools/solid_verifier/solid_dashboard/report/solid_report.log`.

### 2. Через скрипт-обертку

```bash
python run_solid_dashboard.py
```

Скрипт жестко задает `target_dir = ./app` и `config_path = ./solid_config.json`, вызывает внутренний CLI и проксирует его код завершения. Удобен для интеграции в Git-хуки и CI-пайплайны.

### 3. Запуск тестов

Из директории `tools/solid_verifier/`:

```bash
# Все тесты, кроме требующих реального API
pytest -m "not manual" -vv

# Только ручной end-to-end тест с реальным OpenRouter API
# (требует установленной переменной OPENROUTER_API_KEY)
pytest tests/llm/test_open_router_manual.py -m manual -vv -s
```

### Зависимости

Проект использует следующие основные библиотеки (зафиксированы в `tools\solid_verifier\requirements.txt`):

- **[radon](https://pypi.org/project/radon/)** (`>=6.0,<7.0`) — расчет цикломатической сложности и метрик поддерживаемости кода (MI, Halstead).
- **[pydantic](https://pypi.org/project/networkx/)** (`>=2.0,<3.0`) — типизация и валидация схем данных отчётов (`RadonResult`, `CohesionResult` в `schema.py`). Используется как контрактный слой для будущего Report Aggregator.
- **[grimp](https://pypi.org/project/grimp/)** (`>=2.3,<3.0`) — построение и анализ графа импортов на уровне слоёв архитектуры. Используется `ImportGraphAdapter` для расчёта метрик стабильности Мартина (Ca, Ce, Instability), детекции нарушений SDP и SLP.
- **[import-linter](https://pypi.org/project/import-linter/)** (`==2.11`) — проверка соблюдения архитектурных контрактов и изоляции слоев. Версия жестко зафиксирована, так как адаптер парсит неструктурированный текст консольного вывода CLI-утилиты, который может измениться в новых релизах.
- **[lizard](https://pypi.org/project/lizard/)** (`==1.17.10`) — легковесный анализатор, используемый исключительно для извлечения двух побочных метрик: количества параметров (для ISP) и максимальной глубины вложенности. Версия зафиксирована для защиты от изменения внутренней структуры AST-объектов.
- **[pyan3](https://pypi.org/project/pyan3/)** (`>=2.2,<3.0`) — статический анализатор call-graph для современных версий Python (3.10–3.14).
- **[httpx](https://pypi.org/project/httpx/)** (`>=0.27,<2.0`) — HTTP-клиент для реализации шлюза взаимодействия с API провайдеров LLM (OpenRouter/OpenAI).
- **[`jinja2`] (https://pypi.org/project/Jinja2/)** (`>=3.1,<4.0`) — генерация HTML-отчетов (в разработке).
- **[python-dotenv](https://pypi.org/project/python-dotenv/)** (`>=1.0,<2.0`) — безопасная загрузка секретов (например, `OPENROUTER_API_KEY`) из файла `.env` в переменные окружения, исключающая их утечку в систему контроля версий или JSON-конфиги.
- **[networkx](https://pypi.org/project/networkx/)** (`>=3.0,<4.0`) — транзитивная зависимость `grimp` и `pyan3`: обе библиотеки используют её для построения и обхода графов импортов и вызовов. Пинится явно в `pyproject.toml` для гарантии версионной совместимости между двумя потребителями.

Все зависимости жестко зафиксированы в `requirements.txt` на стабильных версиях, совместимых с текущей средой выполнения Python.

## Дорожная карта

- **Ближайшие задачи**

  - Расширение `import_graph_adapter`: детекция циклов на уровне слоев (cycle detection) и полноценное заполнение поля `evidence` для SDP/SLP-нарушений на основе трассы импортов.
  - Доведение LLM-слоя до production-ready: стабилизация промптов и JSON-схемы ответа, снижение числа `parse_failures` до нуля на типичных проектах при сохранении строгого ACL (никакая ошибка модели не должна ломать статический отчёт).
  - Тонкая настройка работы с инфраструктурными классами (`Postgres*Repository`, `ScopusHTTPClient`): выбор между «LLM-only» анализом и полной фильтрацией через усиленный `ClassRole` (`INFRA_SERVICE`/`INFRA_CLIENT`), чтобы контроль OCP/LSP был максимально сфокусирован на доменных слоях.
  - Развитие визуального HTML-дашборда (`generator.py`, `differ.py`) с явным отображением вклада статического слоя, эвристик и LLM для каждого finding.

- **Среднесрочные задачи**

  - Выделение SOLID Verifier в независимый пакет (`pip install solid-verifier`) с сохранением текущей архитектуры адаптеров и LLM-слоя.
  - Интеграция с IDE/VS Code для интерактивного просмотра LSP/OCP-кандидатов и контекстных рекомендаций LLM прямо в редакторе.
  - **Перевод `pyan3_adapter` на режим `--dot`:** замена text-парсера на разбор Graphviz DOT-формата.
  - Расширение `cohesion_adapter` для корректного учёта атрибутов, объявленных через `__slots__`, `dataclasses.field()` и Pydantic-валидаторы — для точной классификации таких классов и исключения ложных нарушений LCOM4.

- **Долгосрочные задачи**

  - Использование LLM не только для OCP/LSP, но и как «менторского слоя» над метриками SRP/DIP: интерпретация высоких LCOM4, нарушенных import-контрактов и сложных функций в виде человекочитаемых советов по рефакторингу, при этом **без** передачи нейросети права переписывать сам отчёт.
